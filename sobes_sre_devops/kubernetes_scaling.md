## Настройка requests и limits

`Requests` и `limits` управляют CPU и Memory, выделяемыми контейнерам в кластере Kubernetes. Они помогают гарантировать, что контейнеры получают необходимые ресурсы для работы, и предотвращают их чрезмерное потребление.

### Что такое requests?

**Requests** задают **минимальное** количество ресурсов, которое контейнеру **гарантировано** будет выделено. Kubernetes Scheduler использует значения requests при назначении Pods на узлы: если узел не может удовлетворить запросы всех контейнеров в Pod, он не будет выбран для размещения этого Pod.

**Пример:**
```yaml
resources:
  requests:
    memory: "64Mi"
    cpu: "250m"  # 250 миллиядер (0.25 vCPU)
```
*В этом примере контейнеру гарантируется 64 MB оперативной памяти и 250 миллиядер процессорного времени.*

### Что такое limits?

**Limits** задают **максимальное** количество ресурсов, которое контейнер может использовать.

*   **Превышение limits по Memory:** Контейнер будет завершён с ошибкой **Out of Memory (OOMKill)**.
*   **Превышение limits по CPU:** Период использования процессорного времени будет ограничен (троттлинг).
    *   ⚠️ **Важно:** Установка limits на CPU может быть плохой практикой, так как при троттлинге замедляется не только этот контейнер, но и все Pods, использующие тот же процессор, что негативно сказывается на производительности всей системы.

**Пример:**
```yaml
resources:
  limits:
    memory: "128Mi"
    cpu: "500m"  # 500 миллиядер (0.5 vCPU)
```
*Контейнер может использовать до 128 MB памяти и 500 миллиядер CPU, но не более.*

### Полный пример манифеста

Рассмотрим полный пример манифеста Pod, который включает как requests, так и limits:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: example-pod
spec:
  containers:
  - name: example-container
    image: nginx
    resources:
      requests:
        memory: "64Mi"
        cpu: "250m"
      limits:
        memory: "128Mi"
        cpu: "500m"
```

Если в вашем Pod несколько контейнеров, `requests` и `limits` устанавливаются **для каждого контейнера** внутри Pod.

### Преимущества использования

Установка `requests` и `limits` положительно влияет на работу кластера:

*   **Эффективное использование ресурсов:** Помогает оптимально распределить ресурсы кластера между Pods, избегая монополизации.
*   **Повышение отказоустойчивости:** Предотвращает случаи нехватки ресурсов для критических Pods (однако помните про осторожность с CPU limits).
*   **Оптимизация автоматического масштабирования:** Позволяет точнее рассчитывать потребности при масштабировании (HPA).

## ResourceQuota: Ограничение ресурсов в Namespace

💡 **ResourceQuota** — это объект Kubernetes, который позволяет ограничить количество ресурсов, доступных в определённом Namespace.

Когда несколько команд работают в одном кластере, одна из них может утилизировать слишком много ресурсов (CPU, Memory, Storage). ResourceQuota помогает это контролировать.

### Как работают ResourceQuota

1.  Разные команды работают в разных Namespaces (это обеспечивается с помощью RBAC).
2.  Администратор создаёт `ResourceQuota` для каждого пространства имён.
3.  Система квот отслеживает использование ресурсов, чтобы оно не превышало установленные жёсткие ограничения.
4.  Если создание или обновление ресурса нарушает ограничение квоты, запрос завершается с кодом состояния HTTP `403 FORBIDDEN`.
5.  Если квота включена для CPU и Memory, пользователи **обязаны** указать `requests` или `limits` для этих значений.

### Пример манифеста и команды

**Пример манифеста ResourceQuota:**
```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: example-resource-quota
  namespace: example-namespace
spec:
  hard:
    requests.cpu: "2"
    requests.memory: 4Gi
    limits.cpu: "4"
    limits.memory: 8Gi
    count/pods: "10"
    count/services: "5"
    count/deployments.apps: "3"
```
Здесь ограничиваются максимальные суммы запросов/лимитов, а также количество объектов в namespace.

**Просмотр квот:**
```bash
kubectl get quota --namespace=example-namespace
kubectl describe quota example-resource-quota --namespace=example-namespace
```

## Pod Topology Spread Constraints (PTSC)

💡 **Pod Topology Spread Constraints** — это механизм Kubernetes, который помогает распределять Pods равномерно по топологическим доменам: зонам доступности, узлам или другим пользовательским топологиям.

### Преимущества и параметры

*   **Повышение отказоустойчивости:** Снижает риск одновременного отказа всех Pods при сбое в одной зоне или узле.
*   **Оптимизация производительности:** Предотвращает перегрузку отдельных узлов и зон.

**Основные параметры:**

*   `maxSkew`: Максимальное отклонение в количестве Pods между топологическими доменами.
*   `topologyKey`: Ключ, определяющий топологию (`topology.kubernetes.io/zone`, `kubernetes.io/hostname`).
*   `whenUnsatisfiable`: Стратегия поведения (`DoNotSchedule` — запретить, `ScheduleAnyway` — разрешить).
*   `labelSelector`: Селектор меток для определения целевых Pods.

### Примеры распределения по зонам и узлам

**Пример 1: Распределение только по зонам доступности:**
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: example-deployment
spec:
  replicas: 4
  selector:
    matchLabels:
      app: example
  template:
    metadata:
      labels:
        app: example
    spec:
      topologySpreadConstraints:
      - maxSkew: 1
        topologyKey: topology.kubernetes.io/zone
        whenUnsatisfiable: DoNotSchedule
        labelSelector:
          matchLabels:
            app: example
      containers:
      - name: example-container
        image: nginx
```
*Pods будут распределены по зонам так, чтобы разница в их количестве между зонами была не более 1.*

**Пример 2: Распределение по зонам И по узлам (рекомендуется):**
Чтобы Pods не оказались все в одной зоне на одном узле, добавляем второе ограничение:
```yaml
spec:
  topologySpreadConstraints:
  - maxSkew: 1
    topologyKey: topology.kubernetes.io/zone   # Равномерно по зонам
    whenUnsatisfiable: DoNotSchedule
    labelSelector:
      matchLabels:
        app: example
  - maxSkew: 1
    topologyKey: kubernetes.io/hostname        # Равномерно по узлам
    whenUnsatisfiable: DoNotSchedule
    labelSelector:
      matchLabels:
        app: example
```
Это гарантирует, что поды будут распределены максимально широко, избегая концентрации как в одной зоне, так и на одном узле.

## Pod Disruption Budget (PDB)

💡 **Pod Disruption Budget (PDB)** — это объект Kubernetes, который помогает контролировать количество Pods приложения, которые могут быть одновременно остановлены в процессе **плановых** событий (обновлений, перезапуска узлов, масштабирования). PDB обеспечивает минимальную доступность приложений.

### Основные параметры

PDB создаётся как отдельный объект и содержит:

*   `minAvailable`: Минимальное количество Pods, которые должны оставаться доступными.
*   `maxUnavailable`: Максимальное количество Pods, которые могут быть одновременно недоступны.
*   `selector`: Селектор меток, определяющий, к каким Pods применяется PDB.

❗️ В манифесте PDB может быть указано **только одно** из полей — `minAvailable` или `maxUnavailable`.

**Пример с `minAvailable`:**
```yaml
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: example-pdb
spec:
  minAvailable: 2
  selector:
    matchLabels:
      app: example
```
*Как минимум 2 пода с меткой `app=example` должны оставаться доступными.*

**Пример с `maxUnavailable`:**
```yaml
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: example-pdb
spec:
  maxUnavailable: 1
  selector:
    matchLabels:
      app: example
```
*Не более одного пода с меткой `app=example` может быть недоступно одновременно.*

### Примеры работы PDB

**Сценарий:** Есть Deployment с 5 подами и PDB с `minAvailable: 3`.

1.  **Обновление Deployment:** Контроллер обновляет поды по одному. Перед удалением каждого пода он проверяет, не нарушит ли это PDB (останется ли 3 доступных?). Если удаление следующего пода нарушит ограничение, обновление приостанавливается до появления нового пода.
2.  **Обновление узла:** Если нужно вывести узел из эксплуатации, Kubernetes проверяет, что после удаления подов с этого узла в кластере останется доступно не менее 3 подов. Если нет, операция будет отложена.

PDB гарантирует, что достаточное количество Pods останется доступным, что помогает избежать деградации сервиса при плановых работах.

## Рекомендации по отказоустойчивости

Чтобы обеспечить высокую доступность и отказоустойчивость кластера (например, **Yandex Managed Service for Kubernetes**), следуйте этим рекомендациям:

1.  **Управляйте обновлениями вручную:**
    Обновляйте кластер и группы узлов только вручную, используя каналы Regular или Stable. Отключите автоматические обновления для мастера и групп узлов.

2.  **Выбирайте правильный тип мастера:**
    Для production-среды используйте **региональный** тип мастера (распределен по зонам), для разработки и тестирования — более дешёвый **зональный**.

3.  **Используйте автоматическое масштабирование групп узлов в нескольких зонах:**
    Создавайте группы узлов с Cluster Autoscaler в нескольких зонах доступности. Это повысит надежность: при сбое одной зоны или узла, новые узлы будут запущены в рабочей зоне.

4.  **Всегда настраивайте requests и limits:**
    Это основа для эффективного планирования и масштабирования.

5.  **Разворачивайте приложения с несколькими репликами в разных зонах:**
    Используйте **Deployment/StatefulSet** с количеством реплик > 1 и механизмом **Pod Topology Spread Constraints** для распределения по зонам.

6.  **Используйте Pod Disruption Budget (PDB):**
    Защитите критичные приложения от деградации во время плановых остановок подов.

7.  **Обеспечьте быстрый запуск приложений:**
    Чем быстрее запускается и завершается ваш Pod, тем быстрее кластер может реагировать на сбои и масштабирование.

8.  **Настройте корректные пробы (Probes):**
    `ReadinessProbe` и `LivenessProbe` помогают Kubernetes понимать состояние приложения и своевременно направлять трафик только на готовые экземпляры.
