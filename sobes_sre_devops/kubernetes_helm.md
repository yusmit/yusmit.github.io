
# Структура и создание Helm-чартов

Helm-чарт — это набор файлов и директорий, организованных определённым образом для упаковки и развёртывания Kubernetes-приложений.

## Минимальная структура чарта

Для работы простейшего чарта необходимы три компонента:

- **`Chart.yaml`** — метаданные чарта
- **`values.yaml`** — значения по умолчанию
- **`templates/`** — директория с шаблонами манифестов

Остальные файлы (`README.md`, `LICENSE`, `charts/`, `values.schema.json`) опциональны.

## Полная структура чарта

```
mychart/
├── Chart.yaml                # метаданные чарта (обязательный)
├── values.yaml               # значения по умолчанию (обязательный)
├── templates/                # шаблоны манифестов (обязательный)
│   ├── deployment.yaml
│   ├── service.yaml
│   ├── _helpers.tpl          # вспомогательные шаблоны
│   └── NOTES.txt             # инструкции после установки
├── charts/                   # зависимости (сабчарты)
├── values.schema.json        # схема валидации значений
├── LICENSE                   # лицензия
└── README.md                 # документация
```

## Основные компоненты

### Chart.yaml — метаданные чарта

Обязательный файл с информацией о чарте:

```yaml
apiVersion: v2           # версия API чарта (v2 для Helm 3)
name: mychart            # имя чарта
version: 0.1.0           # версия чарта (SemVer2)
description: A Helm chart for Kubernetes  # описание (необязательно)
appVersion: "1.16.0"     # версия приложения (необязательно)
```

### values.yaml — значения по умолчанию

Содержит переменные, которые пользователь может переопределить при установке:

```yaml
replicaCount: 2

image:
  repository: nginx
  tag: "1.16.0"

service:
  port: 80
```

### templates/ — шаблоны манифестов

Директория с Go-шаблонами, которые рендерятся в Kubernetes-манифесты. Пример `templates/deployment.yaml`:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ .Release.Name }}-deployment
spec:
  replicas: {{ .Values.replicaCount }}
  template:
    spec:
      containers:
        - name: {{ .Chart.Name }}
          image: {{ .Values.image.repository }}:{{ .Values.image.tag }}
```

## Дополнительные файлы

### values.schema.json — валидация значений

JSON-схема для проверки входных данных:

```json
{
  "$schema": "https://json-schema.org/draft-07/schema#",
  "properties": {
    "port": {
      "description": "Port",
      "minimum": 0,
      "type": "integer"
    },
    "protocol": {
      "type": "string"
    }
  },
  "required": ["protocol", "port"],
  "type": "object"
}
```

### NOTES.txt — инструкции после установки

Файл в `templates/`, содержимое которого выводится после успешной установки:

```
Thank you for installing {{ .Chart.Name }}.

Your application is available at:
  http://{{ .Values.service.name }}:{{ .Values.service.port }}

Enjoy!
```

### charts/ — зависимости

Директория для хранения сабчартов (зависимостей). Сюда помещаются другие чарты, от которых зависит текущий.

## Создание чарта

### Генерация скелета

```bash
helm create mychart
```

Команда создаст структуру:

```
mychart/
├── charts/
├── Chart.yaml
├── templates/
│   ├── deployment.yaml
│   ├── _helpers.tpl
│   ├── hpa.yaml
│   ├── ingress.yaml
│   ├── NOTES.txt
│   ├── serviceaccount.yaml
│   ├── service.yaml
│   └── tests/
│       └── test-connection.yaml
└── values.yaml
```

## Проверка чарта

### 1. Линтинг — проверка синтаксиса

```bash
helm lint mychart/
```

Проверяет YAML-разметку и структуру чарта.

### 2. Проверка шаблонов

```bash
helm template mychart/
```

Показывает результат рендеринга шаблонов без установки.

### 3. Пробный запуск (dry-run)

```bash
helm install my-release mychart/ --dry-run
```

Имитирует установку в кластер, проверяет, примет ли Kubernetes манифесты.

### Упаковка чарта

```bash
helm package mychart/
# Результат: mychart-0.1.0.tgz
```

## Параметризация шаблонов

### Встроенные объекты Helm

| Объект | Описание |
|--------|----------|
| `.Values` | Значения из `values.yaml` и `--set` |
| `.Release` | Информация о релизе (имя, версия и т.д.) |
| `.Chart` | Метаданные из `Chart.yaml` |
| `.Files` | Доступ к файлам чарта |
| `.Capabilities` | Информация о кластере |

### Переменные и подстановки

```yaml
name: {{ .Release.Name }}-deployment
replicas: {{ .Values.replicaCount }}
image: {{ .Values.image.repository }}:{{ .Values.image.tag }}
```

### Функции и конвейеры (pipeline)

```yaml
replicas: {{ .Values.replicaCount | default 1 }}
image: {{ .Values.image.tag | default "latest" | quote }}
```

Популярные функции:
- `default` — значение по умолчанию
- `quote` — заключает в кавычки
- `upper` / `lower` — регистр
- `indent` / `nindent` — добавляет отступы

### Условные конструкции

```yaml
metadata:
  name: {{ .Release.Name }}-deployment
  {{- if .Values.nameLabel }}
  labels:
    name: {{ .Values.nameLabel }}
  {{- end }}
```

`{{-` удаляет пробелы слева, `-}}` — справа.

### Циклы (range)

```yaml
containers:
  {{- range .Values.containers }}
  - name: {{ .name }}
    image: {{ .image }}
  {{- end }}
```

Внутри цикла `.` ссылается на текущий элемент.

## Лучшие практики

### Именование
- Имя чарта: **строчные буквы и цифры**, слова через дефис (`nginx`, `argo-cd`)
- Переменные: **camelCase** с первой строчной (`replicaCount`, `revisionHistoryLimit`)
- Файлы шаблонов: **дефисная нотация** (`deployment.yaml`, `service.yaml`)

### Версионирование
- Версия чарта: **SemVer2** (`MAJOR.MINOR.PATCH`)
- При изменении MAJOR сбрасываются MINOR и PATCH
- При изменении MINOR сбрасывается PATCH

### values.yaml
- **Предпочитать плоскую структуру** вложенной (проще для пользователей)
- **Документировать все переменные**:
  ```yaml
  # serverPort is the HTTP listener port
  serverPort: 9191
  ```
- **Явно указывать типы**: строки в кавычках, числа без кавычек
- Для больших чисел использовать строки: `number: "123456789"`

### templates
- Один ресурс — один файл
- Имена файлов отражают тип ресурса (`deployment.yaml`, `svc.yaml`)
- Отступы — **2 пробела** (не табуляция)

### Документирование
- `README.md` — описание, установка, настройка
- `NOTES.txt` — инструкции после установки
- `LICENSE` — лицензионное соглашение

## Helm в production: расширенная теория

### Как правильно позиционировать Helm

`Helm` полезно описывать не как "магическую установку в Kubernetes", а как инструмент упаковки и управления релизом:
- chart задает структуру приложения;
- values задают параметры окружения;
- release хранит состояние установки;
- rollback позволяет быстро вернуться к прошлой версии релиза.

### Что должно быть в хорошем production-chart

1. Ясная структура values:
- образ;
- ресурсы;
- probes;
- ingress;
- affinity/tolerations;
- env;
- volumes;
- service account;
- network policy или связанные флаги интеграции.

2. Валидация входных значений:
- `values.schema.json`;
- дефолты без сюрпризов;
- понятные required-поля.

3. Минимум "магии":
- не прятать критические решения в сложные вложенные `if`;
- не плодить дубли через копипасту шаблонов;
- не превращать chart в универсальный конструктор на все случаи жизни.

### Helm и GitOps (`FluxCD`)

Если `Helm` используется вместе с GitOps, на интервью полезно проговаривать связку:
- `Helm` отвечает за шаблонизацию и структуру релиза;
- `FluxCD` удерживает желаемое состояние из Git;
- изменения не должны ехать в кластер напрямую с ноутбука инженера.

Хорошая инженерная модель:
1. Изменение values или chart попадает в Git.
2. Проходит review и валидацию.
3. Flux подтягивает изменение и делает reconciliation.
4. Откат делается через Git или rollback релиза, а не через ручное "починил в кластере".

### Шире про delivery-модель Helm

`Helm` можно использовать в нескольких режимах:
- как локальный инструмент релиза;
- как часть CI/CD-пайплайна;
- как часть GitOps-модели через `HelmRelease` или аналогичный controller pattern;
- как способ упаковки платформенных компонентов, а не только прикладных сервисов.

Важно понимать, что сам по себе `Helm` не решает:
- drift management;
- policy enforcement;
- секреты;
- безопасную модель доступа.

Он хорошо решает шаблонизацию и управление релизом, но должен жить в более широкой инженерной модели.

### Helm hooks, dependencies и release lifecycle

#### Hooks
- нужны для действий до/после install/upgrade/delete;
- полезны, но легко превращаются в скрытую магию, если в hooks унести критичную бизнес-логику релиза;
- полезно проговаривать, что hooks должны быть минимальными и предсказуемыми.

#### Dependencies
- позволяют собирать решение из нескольких chart dependencies;
- удобны для платформенных компонентов, но усложняют версионирование и диагностику;
- важно понимать, где dependency уместна, а где лучше явная композиция через GitOps или отдельные релизы.

#### Release lifecycle
- install;
- upgrade;
- rollback;
- uninstall;
- drift между ожидаемым и фактическим состоянием, если вокруг Helm нет контролируемой delivery-модели.

### Частые anti-patterns в Helm

- использовать `latest` для образов;
- не задавать `requests/limits`;
- не задавать probes;
- смешивать несколько логических компонентов в одном шаблоне без необходимости;
- хранить секреты прямо в values без нормального процесса работы с секретами;
- делать chart, который невозможно отрендерить и проверить вне кластера.

### Как отлаживать проблемы Helm-релиза

#### Если релиз не ставится
1. `helm lint`
2. `helm template`
3. `helm upgrade --install --dry-run --debug`
4. Проверить rendered manifest на ошибки структуры, API version, values и шаблонов.

#### Если релиз поставился, но приложение не работает
1. Проверить, какие манифесты реально отрендерились.
2. Проверить `Deployment/StatefulSet`, `Events`, `describe`, `logs`.
3. Проверить probes, ресурсы, env, секреты, сервисную связность.
4. Проверить, нет ли конфликта между chart values, policy и admission-контролем.

### Вопросы для собеседования

#### Вопрос: Чем `helm template` отличается от `helm upgrade --install --dry-run --debug`?
Ответ:
- `helm template` просто рендерит манифесты локально.
- `dry-run --debug` ближе к реальной установке: показывает, как Helm соберет релиз и какие значения реально применятся.
- Для зрелого ответа полезно сказать, что оба режима нужны, но решают немного разные задачи.

#### Вопрос: Зачем chart нужен `values.schema.json`?
Ответ:
- Чтобы ловить ошибки конфигурации до установки.
- Чтобы не принимать в релиз произвольные и случайно сломанные значения.
- Это особенно важно в больших командах и при GitOps-подходе.

#### Вопрос: Когда Helm становится проблемой, а не решением?
Ответ:
- Когда chart перегружен логикой.
- Когда из него нельзя понять итоговый манифест.
- Когда values бесконтрольны и не валидируются.
- Когда Helm используется как замена нормальной инженерной модели изменений.

#### Вопрос: Когда chart dependency - это хорошая идея, а когда нет?
Ответ:
- Хорошая идея, когда есть действительно переиспользуемый компонент с понятным lifecycle.
- Плохая идея, когда зависимости скрывают слишком много неявной логики и усложняют отладку релиза.
- В зрелой платформе важно не только "можно ли подключить dependency", но и кто потом будет сопровождать ее версионирование и совместимость.
